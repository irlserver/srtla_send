T12 — CONN_TIMEOUT investigate + reconcile (srtla-send-rs)
==========================================================

DECISION: (b) Accidental drift — align the Rust sender's CONN_TIMEOUT from 5 s to
15 s to match the bonding receiver and the C sender. Pin the value and the
sender-vs-receiver relationship with a test.

------------------------------------------------------------------
THE DIVERGENCE (verified)
------------------------------------------------------------------
  Rust sender   : CONN_TIMEOUT        = 5   (src/protocol/constants.rs:32)  [BEFORE]
  C   sender    : SENDER_CONN_TIMEOUT = 15  (srtla/src/sender_logic.h:66)
  C   receiver  : CONN_TIMEOUT        = 15  (srtla/src/receiver_config.h:28)

At 5 s the Rust sender declared an uplink "failed" 3x sooner than the receiver
holds it — the receiver keeps the link for 15 s and keeps echoing keepalives the
whole time. This is a behavioral timing divergence only; NOT a wire-protocol break.

------------------------------------------------------------------
GIT ARCHAEOLOGY — intent of the Rust 5 s  (full log: task-12-gitlog.txt)
------------------------------------------------------------------
  - 7de8b8d "initial rust version"  : CONN_TIMEOUT = 4   (upstream irlserver)
  - a003e06 "refactor: split up ... file": bumped 4 -> 5 (upstream, incidental to a
    file-split refactor — NOT a deliberate timeout-tuning commit)
  - c795119 "Refactoring"           : value 5 preserved, const MOVED protocol.rs ->
    protocol/constants.rs
  - git merge-base --is-ancestor 7de8b8d 80cd0c4  => TRUE
        => ALL CONN_TIMEOUT history (4, then 5) is UPSTREAM, before the fork base
           (fork start = upstream 80cd0c4; first CERALIVE commit = ad8730e bootstrap)
  => CeraLive never deliberately chose 5 s. It was inherited verbatim from upstream.

Meanwhile, on the C side, CeraLive DELIBERATELY widened the sender 4 -> 15 s with a
documented rationale (srtla/src/sender_logic.h:50-65, Task 9 case d
"SenderFalselyDownsAliveLinkOnSubReceiverTimeoutGap"): a sub-receiver-timeout sender
needlessly re-registers and resets the window to WINDOW_MIN on a link that is merely
mid radio-stall (the "server in another country" false link-down), while the receiver
and the physical link are fine.

CONCLUSION: this is accidental drift. The C/receiver pair was tuned to 15 s on
purpose; the Rust fork's 5 s is an un-reconciled upstream leftover.

------------------------------------------------------------------
WHY WIDENING DOES NOT SLOW FAILOVER (same argument the C code makes)
------------------------------------------------------------------
Real dead-link detection in the Rust sender is timeout-INDEPENDENT:
  1. Hard send failure: a failed batch flush (sendto error on a torn-down link)
     immediately marks the link for recovery in <1 s
       — src/sender/packet_handler.rs:348  flush_batch() -> mark_for_recovery()
  2. Quality-score deselection: a struggling/lossy link is deselected by the
     EDPF/BLEST/IoDS scorer within ~1 s
       — commit 35c4d3b "fix(srtla_send): demote lossy links continuously, never
         hard-kill"
CONN_TIMEOUT only governs when an established-but-silent link is reset + re-registered
(src/sender/housekeeping.rs:52 is_timed_out() -> reconnect()). A shorter value adds
FALSE link-downs during radio stalls; it does not add real failover speed.
